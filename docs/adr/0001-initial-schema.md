# ADR-0001: Начальная схема данных

- **Статус:** принято
- **Дата:** 2026-05-16
- **Версия схемы:** 1

## Контекст

На старте проекта нужен единый формат для хранения сведений о фильмах,
людях, студиях, символах/мотивах и референсах. Требования:

1. Данные читаемы человеком и редактируемы без специальных инструментов.
2. Данные версионируемы через git: один файл — одна сущность.
3. Схема валидируется автоматически в CI.
4. Связи между сущностями выражаются как ссылки на slug-и (не на числовые id),
   стабильные после публикации.
5. Возможна постепенная миграция при росте требований без переписывания
   уже размеченных файлов.

## Решение

Принят формат **YAML**, по одному файлу на сущность, в директориях
`films/`, `people/`, `studios/`, `motifs/`, `references/`. Слаги в латинице
(kebab-case) и совпадают с именем файла.

Контролируемые словари (страны, республики, жанры, роли, категории мотивов,
типы референсов, статусы цензуры, языки) хранятся в `vocabularies/` как YAML
со списком `values` (см. соответствующие файлы). Новые значения добавляются
только через PR в словарь.

Каждая сущность содержит поле `schema_version: 1`. При несовместимых изменениях
выпускается миграционный скрипт в `soviet-cinema-tools/migrations/` и
бампится `schema_version`.

Pydantic-модели (см. `soviet-cinema-tools/validators/schemas.py`) — источник
истины относительно формы данных. Валидатор проверяет:

- соответствие моделям;
- совпадение `id` с именем файла;
- уникальность `id` внутри каждой папки;
- разрешение перекрёстных ссылок (director/cast/screenwriter → people, studio →
  studios, source_film → films);
- значения полей, опирающихся на словари (country, genre, role, motif category,
  reference kind, censorship_status, republic, language).

## Альтернативы

1. **JSON вместо YAML.** Отвергнуто: хуже читается, тяжелее редактируется
   вручную (комментарии, многострочные строки).
2. **TOML.** Отвергнуто: формат менее распространён в гуманитарных проектах,
   слабая поддержка вложенных массивов с метаданными.
3. **Единый монолитный JSON/CSV.** Отвергнуто: плохо ложится на git и
   человеческое ревью.
4. **СУБД как источник истины.** Отвергнуто на этом этапе: усложняет вход
   контрибьюторам и противоречит принципу «данные отделены от приложения».
   SQLite остаётся как рантайм-кэш, генерируемый на сборке.

## Последствия

- Любое нетривиальное изменение схемы — отдельный ADR + миграция.
- Расширение словарей не требует бампа `schema_version`, но требует PR в
  `vocabularies/`.
- Slug — публичный идентификатор. Переименование slug-а считается
  ломающей операцией и требует отдельной процедуры (редирект, миграция эссе).
